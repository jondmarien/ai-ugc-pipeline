DirtyDecrypt (CVE-2026-31635) is an LPE in rxgk_decrypt_skb missing a copy-on-write guard. Only hits distros with CONFIG_RXGK enabled: Fedora, Arch Linux, openSUSE Tumbleweed.

Zellic and V12 reported it May 9. Maintainers said: duplicate, already patched in mainline. No CVE. No advisory. No coordinated disclosure. Researchers forced public when they realized the fix was silent.

The bug: rxgk_decrypt_skb decrypts incoming socket buffer on receive. Memory pages shared with page cache of other processes. Copy-on-write should protect shared pages. The guard was missing. Data writes to privileged process memory or page cache of /etc/shadow, /etc/sudoers, SUID binaries.

No embargo broke. No third party weaponized a commit. V12 chose to release the PoC voluntarily once they confirmed the fix was already in mainline, with the explicit note: We're releasing it now since it's patched. That choice is what made the gap visible.

Silent patches without CVEs leave defenders blind. Researchers who report responsibly get duplicate responses.

Kernel developers now reviewing emergency killswitch: disable vulnerable function at runtime until real patch arrives. Rocky Linux debuted opt-in security repository for accelerated fixes.

A silent patch is not a fix for defenders who do not know it exists. The killswitch and security repos are symptoms of a disclosure process that leaves gaps.
